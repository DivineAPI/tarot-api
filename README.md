## What is the Tarot API?

The **Tarot API** by DivineAPI is a complete suite of 23+ REST endpoints for tarot and divination content. It covers every major tarot product category developers build: one-card daily pulls, yes-or-no questions, past-present-future three-card spreads, deep love readings, compatibility scores, career guidance, angel and Egyptian readings, dream interpretations, and mystical specialties like coffee cup and past-life readings.

Every reading returns rich text (multi-paragraph interpretations by professional tarot readers) plus hosted card images ready to embed in a webpage or mobile app. No SDK, no client-side deck library: one API key, one call, a ready-to-render response. Content is available in 25 languages.

Part of the broader [DivineAPI platform](https://github.com/DivineAPI/astrology-api) (200+ astrology, horoscope, tarot and numerology endpoints).

## Walkthrough: Daily Tarot

The simplest endpoint in the suite and a great first call. No personal data required, just your API key. Returns one card with its category (upright or reversed), interpretations across career, love and finance, and two image URLs (the card face for upright and reversed variants).

```http
POST https://astroapi-5.divineapi.com/api/v2/daily-tarot
```

Authenticate with a Bearer token in the `Authorization` header **and** pass `api_key` in the request body (both required).

### Request body

| Parameter | Type | Required | Description | Example |
|---|---|:---:|---|---|
| `api_key` | string | ✓ | Your DivineAPI key | `YOUR_API_KEY` |
| `lan` | string | - | Language code (default `en`) | `en` |

That's it. No sign, no date, no personal details. Each call draws a fresh card.

Full docs → **[developers.divineapi.com/horoscope-and-tarot-api/daily-tarot](https://developers.divineapi.com/horoscope-and-tarot-api/daily-tarot)**

## What's in a reading?

Most endpoints in this suite share a similar response shape, so you can build one card-rendering component and reuse it across features.

- **`card`** (or card name equivalent): the name of the drawn card or spread element
- **`category`**: `Upright` or `Reverse`, drives the tone of the interpretation
- **Interpretive text fields**: usually `career`, `love`, `finance`, sometimes `health`, `emotions`, or long-form `reading` blocks
- **Images**: `image` (primary) and `image2` (the opposite orientation) as public JPG URLs, safe to embed directly

Different endpoints specialize: love-related endpoints usually return only a `love` block (and sometimes a compatibility percentage), career-focused endpoints return a career-themed long read, three-card spreads return three card objects. Check each endpoint's sample response on the docs site for the exact schema.

## Code examples

### cURL

```bash
curl -X POST "https://astroapi-5.divineapi.com/api/v2/daily-tarot" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  --data-urlencode "api_key=YOUR_API_KEY"
```

### Python (requests)

```python
import requests

url = "https://astroapi-5.divineapi.com/api/v2/daily-tarot"

headers = {
    "Authorization": "Bearer YOUR_API_KEY",
    "Content-Type": "application/x-www-form-urlencoded",
}

payload = {
    "api_key": "YOUR_API_KEY",
}

response = requests.post(url, headers=headers, data=payload)
card = response.json()["data"]
print(f"{card['card']} ({card['category']})")
print(card["career"])
```

### Go (net/http)

```go
package main

import (
    "encoding/json"
    "fmt"
    "io"
    "net/http"
    "net/url"
    "strings"
)

type Response struct {
    Success int `json:"success"`
    Data    struct {
        Card     string `json:"card"`
        Category string `json:"category"`
        Career   string `json:"career"`
        Love     string `json:"love"`
        Finance  string `json:"finance"`
        Image    string `json:"image"`
    } `json:"data"`
}

func main() {
    endpoint := "https://astroapi-5.divineapi.com/api/v2/daily-tarot"

    form := url.Values{}
    form.Set("api_key", "YOUR_API_KEY")

    req, _ := http.NewRequest("POST", endpoint, strings.NewReader(form.Encode()))
    req.Header.Set("Authorization", "Bearer YOUR_API_KEY")
    req.Header.Set("Content-Type",  "application/x-www-form-urlencoded")

    resp, err := http.DefaultClient.Do(req)
    if err != nil {
        panic(err)
    }
    defer resp.Body.Close()

    body, _ := io.ReadAll(resp.Body)
    var r Response
    json.Unmarshal(body, &r)
    fmt.Printf("Today's card: %s (%s)\n", r.Data.Card, r.Data.Category)
    fmt.Println(r.Data.Career)
}
```

## FAQ

**Are readings deterministic or random?**
Random. Each call to Daily Tarot, Yes or No Tarot, Past-Present-Future and most reading endpoints draws a fresh card (or spread) from the deck. That's what makes it feel like a real tarot experience. Love Compatibility and similar inputs-driven endpoints are deterministic given identical inputs.

**How do I lock in one card per user per day?**
Wrap the API with your own cache keyed by `(user_id, date)`. On first call of the day, store the response in your DB or Redis; subsequent calls that day return your cached copy. This also saves quota.

**What's the difference between Daily Tarot and Yes or No Tarot?**
Daily Tarot draws one card and returns interpretations across career, love and finance plus images. Yes or No Tarot answers a binary question with a card whose orientation maps to a yes/no/maybe verdict.

**Upright vs Reversed, what does the `category` field mean?**
Traditional tarot cards can be drawn right-side-up (`Upright`) or upside-down (`Reverse`). Reversed cards usually carry a contrasting, obstructed, or shadow meaning compared to their upright version. Both interpretations are provided in the response (and both image orientations are available via `image` and `image2`).

**Are card images free to display in commercial apps?**
Yes. The image URLs returned (`image`, `image2`, plus any card art in specialty readings) are hosted by DivineAPI and safe to embed directly in your app or website, including commercial ones.

**What's the difference between this repo and the per-reading repos (`daily-tarot`, `yes-or-no-tarot`, etc.)?**
Same underlying endpoints. The per-reading repos are single-endpoint landing pages targeting specific SEO keywords (e.g. "yes or no tarot api"). This `tarot-api` repo is the umbrella showing all 23 reading endpoints in one place. Pick whichever reads better for the audience you're linking.

**Can I request readings in languages other than English?**
Yes. Pass `lan=` with a supported language code to get the full response translated. 25 languages are supported.

**Does the API use a specific tarot deck?**
The imagery is based on classical Rider-Waite-Smith style deck traditions, rendered with DivineAPI's own art. Card meanings follow traditional tarot interpretation conventions.
